# Planetary Infrastructure Framework

**Planetary Infrastructure Framework (PIF)** — это мод-фреймворк для Stellaris, который перестраивает слой планетарной инфраструктуры так, чтобы моддерам было проще делать точечные изменения и меньше конфликтовать друг с другом.

Фреймворк построен на vanilla-файлах Stellaris **4.3.7** и сохраняет vanilla-поведение игры.

Для обычного игрока PIF сам по себе не должен менять gameplay. Его основное назначение — быть зависимостью и технической основой для других модов.

## Идея проекта

Во время работы над модификациями для Stellaris я постоянно сталкивался с одной и той же проблемой: даже небольшая правка часто требует переписывать большой vanilla-объект целиком.

Если нужно изменить стоимость здания, количество jobs, условие доступности, вес назначения pop-а на job или какой-то отдельный modifier, приходится копировать и переопределять весь объект. Когда несколько модов делают это одновременно, они начинают конфликтовать даже в тех случаях, когда меняют совершенно разные части логики.

В результате появляются сложные compatibility patches, а простые балансные модификации становятся тяжелее в поддержке, чем должны быть.

PIF появился как попытка облегчить эту работу: сузить места вероятных конфликтов и вынести значимые значения в отдельные переменные, чтобы их можно было менять без затрагивания больших объектов.

Основная идея фреймворка:

- крупные vanilla-объекты разбиваются на небольшие смысловые части;
- каждая такая часть выносится в отдельный inline script;
- значимые числовые и некоторые логические значения выносятся в scripted variables;
- моддер может менять конкретный аспект объекта, не переписывая весь объект целиком.

Иными словами, PIF создаёт промежуточный слой между vanilla Stellaris и модами, которые хотят изменять планетарную инфраструктуру.

## Назначение

Planetary Infrastructure Framework предназначен прежде всего для моддеров, которые хотят изменять планетарную инфраструктуру Stellaris более аккуратно и совместимо.

Фреймворк покрывает следующие типы объектов:

- `districts`;
- `zones`;
- `zone_slots`;
- `buildings`;
- `pop_jobs`.

Каждый объект сохраняет своё vanilla-поведение, но его содержимое раскладывается по смысловым категориям.

Например, для buildings можно отдельно менять:

- стоимость и содержание;
- jobs;
- условия доступности;
- lifecycle-логику;
- planet modifiers;
- country modifiers;
- pop/job modifiers;
- AI-логику.

Для jobs можно отдельно менять:

- производство и upkeep;
- веса назначения pop-ов;
- promotion/demotion;
- automation flags;
- top-level job flags;
- planet/country/pop modifiers;
- overlord resources.

Это позволяет создавать небольшие моды, которые затрагивают только нужный слой логики, а не весь vanilla-объект целиком.

## Что делает фреймворк

PIF:

- переопределяет vanilla-объекты планетарной инфраструктуры;
- сохраняет vanilla-поведение;
- разбивает объекты на маленькие смысловые inline scripts;
- выносит значимые значения в scripted variables;
- создаёт стабильные точки расширения для других модов;
- уменьшает вероятность конфликтов между модами, построенными поверх PIF.

Фреймворк не является балансным модом. Его цель — не изменить игру, а дать удобную техническую основу для других модификаций.

## Совместимость

PIF принципиально несовместим с модами, которые напрямую затрагивают те же vanilla-файлы, на основе которых построен фреймворк.

Список затронутых vanilla-файлов и объектов находится здесь:

[docs/AFFECTED_VANILLA_FILES.md](docs/AFFECTED_VANILLA_FILES.md)

Если другой мод напрямую переопределяет объекты из этих файлов, он, скорее всего, будет конфликтовать с PIF.

Важно понимать идею фреймворка: PIF не пытается быть совместимым с любым модом, который переписывает те же vanilla-объекты. Его цель — создать основу, на которой другие моды смогут строиться так, чтобы конфликтовать друг с другом значительно реже.

То есть:

- моды, напрямую переписывающие те же `districts`, `zones`, `zone_slots`, `buildings` или `pop_jobs`, могут конфликтовать с PIF;
- моды, использующие PIF inline scripts и PIF variables, должны иметь меньшую зону конфликтов;
- два PIF-based мода всё ещё могут конфликтовать, если меняют одну и ту же маленькую часть одного и того же объекта, но им не нужно конфликтовать из-за всего объекта целиком.

PIF обычно должен загружаться перед модами, которые построены поверх него. Моды, напрямую переопределяющие те же vanilla-объекты, могут конфликтовать независимо от порядка загрузки.

## Как использовать PIF в других модах

PIF предоставляет два основных механизма для моддеров.

### Category inline scripts

Каждый объект разбит на смысловые категории.

Например, building может иметь отдельные inline scripts для:

```txt
availability
economy
lifecycle
jobs
economic_modifiers
planet_state
ai
```

Job может иметь отдельные inline scripts для:

```txt
availability
economy
pop_config
swap
planet_state
country_modifiers
pop_modifiers
economic_modifiers
```

Мод может переопределить только нужный inline script, не копируя весь объект.

### Scripted variables

Значимые параметры вынесены в variables.

Например:

- стоимость;
- upkeep;
- производство;
- количество jobs;
- job weights;
- promotion/demotion time;
- planet/country/pop modifiers;
- важные top-level flags у jobs.

Это позволяет делать балансные правки через переменные, не переписывая сами объекты.

## Структура репозитория

Репозиторий состоит из трёх основных частей:

```txt
mod/
docs/
build-scripts/
```

`mod/` содержит сгенерированные игровые файлы.

`docs/` содержит пользовательскую и техническую документацию.

`build-scripts/` содержит инструменты анализа, генерации и валидации, с помощью которых фреймворк собирается из vanilla-данных.

## Версия Stellaris

PIF построен на vanilla-файлах Stellaris **4.3.7**.

В descriptor используется:

```txt
4.3.*
```

Предполагаемый диапазон совместимости — ветка Stellaris 4.3, фактическая baseline-версия для генерации — **4.3.7**.

## Обратная связь

Я буду благодарен за сообщения о найденных ошибках, проблемах совместимости и любых случаях, когда PIF неожиданно меняет vanilla-поведение.

Особенно полезны:

- ошибки, связанные с файлами PIF;
- примеры конфликтов с другими модами;
- случаи, где объект ведёт себя иначе, чем в vanilla;
- предложения по улучшению структуры inline scripts или variables.

PIF создавался как практический инструмент для моддеров, поэтому обратная связь по реальному использованию особенно важна.
